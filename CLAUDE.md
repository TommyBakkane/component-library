# Component Library — Agent Rules

## Project overview

React component library published as an npm package to GitHub Packages (`@tommybakkane/component-library`). Consumers install it, import CSS once, and use components. The library builds to `dist/` as dual ESM + CJS with bundled `style.css`.

---

## File structure

Each component lives in its own folder with three files:

```
src/components/<name>/
  <name>.tsx
  <name>.module.css
  <name>.stories.tsx
```

Layout primitives follow the same pattern under `src/layout/<name>/`.

---

## Compound components

Some components use the compound pattern — `Object.assign` attaches sub-components to the root export so consumers use one import with dot-notation:

```tsx
import { Input } from '@tommybakkane/component-library';

<Input.Field error="Required">
  <Input.Label>Email</Input.Label>
  <Input type="email" />
  <Input.Error />
</Input.Field>
```

Rules for compound components:
- The root export (`Input`) is the primary element (the `<input>` itself), not a wrapper
- Sub-components are attached via `Object.assign(RootComponent, { Field, Label, Hint, Error })`
- `Field` provides a React context (`createContext`) with `id` (from `useId()`), `error`, and `disabled`
- Sub-components that need context call a `useField()` hook that throws if used outside `Field`
- `id` is generated by `useId()` in `Field` and wired automatically: `Label` gets `htmlFor`, the input gets `id` and `aria-describedby`/`aria-invalid`, `Error` gets `id` for the describedby reference
- Export all prop interfaces (`FieldProps`, `LabelProps`, etc.) individually — consumers need them

---

## TypeScript rules

- **Name and export every prop interface** — never use an anonymous or local-only `interface Props`. Name it after the component: `ButtonProps`, `FlexProps`, etc.
- **Always `export` the prop interface** so consumers can extend or reference it.
- **Interactive/HTML components** extend the native element via `React.ComponentPropsWithRef<'element'>`. Layout-only components (no DOM forwarding) can use a plain interface.
- **No `React` import** — the project uses the automatic JSX transform (`react-jsx`).

```tsx
// Correct
export interface ButtonProps extends React.ComponentPropsWithRef<'button'> {
  variant?: 'primary' | 'secondary';
}

// Wrong — local, unnamed, not exported
interface Props { ... }
```

## Styling rules

- **CSS Modules only** — one `.module.css` per component, imported as `styles`.
- **CSS custom properties for all values** — colours, spacing, sizing, and radii must come from tokens (see below). No hardcoded hex values or pixel literals in component CSS except in tokens files.
- **No inline styles** unless the value is genuinely dynamic and cannot be expressed as a CSS class (e.g., `gridTemplateColumns: repeat(${cols}, 1fr)`).
- **No Tailwind, no CSS-in-JS, no `style` props** on components.
- **Class composition** — use the array filter/join pattern. Do not install `clsx`.

```tsx
className={[styles.base, styles[variant], className].filter(Boolean).join(' ')}
```

## Design tokens

All base tokens live in `src/tokens/base.css` as CSS custom properties on `:root`:

| Category | Prefix | Scale |
|----------|--------|-------|
| Colours | `--color-*` | primary, danger, success, warning, info, border, text, text-muted |
| Spacing | `--spacing-*` | xs, sm, md, lg, xl |
| Radius | `--radius-*` | xs, sm, md, lg, xl, full |
| Font size | `--font-size-*` | xs, sm, md, lg, xl |
| Component height | `--size-*` | xs, sm, md, lg, xl |
| Icon size | `--icon-*` | xs, sm, md, lg |

Component-specific tokens (e.g., `--button-bg`, `--button-radius`) are defined in `src/tokens/<component>.css` and referenced in the component's module CSS. When adding a new component that needs overrideable tokens, add a `src/tokens/<name>.css` file and import it in `src/index.ts`.

---

## Barrel (`src/index.ts`)

Every new component and prop type **must** be added to `src/index.ts` before it is considered done. The file also imports all token CSS files as side effects so they are bundled into `dist/style.css`.

Pattern for each addition:

```ts
import './tokens/<name>.css'; // only if there is a tokens file

export { ComponentName } from './components/<name>/<name>';
export type { ComponentNameProps } from './components/<name>/<name>';
```

## Storybook

Every component must have a `.stories.tsx` file using the `Meta` + `StoryObj` pattern:

```tsx
import type { Meta, StoryObj } from '@storybook/react-vite';
import { Button } from './button';

const meta = {
  title: 'Components/Button',
  component: Button,
  tags: ['autodocs'],
  argTypes: { ... },
  args: { ... },
} satisfies Meta<typeof Button>;

export default meta;
type Story = StoryObj<typeof meta>;

export const Primary: Story = {};
```

- Always include `tags: ['autodocs']`.
- Story title hierarchy: `Components/Name` for components, `Layout/Name` for layout primitives.
- Add individual stories for each variant/state. Use a `render` function for composite "All variants" stories.

---

## Accessibility

- Use `aria-busy` on loading states.
- Use `aria-hidden="true"` on decorative icon wrappers.
- Default `type="button"` on all `<button>` elements to prevent accidental form submission.
- Use `data-*` attributes to drive CSS state (e.g., `data-loading`) instead of adding/removing classes.

## Build & publish

```sh
npm run build       # produces dist/
npm run storybook   # dev server on :6006
npm run test        # vitest (Storybook stories via Playwright/Chromium)
npm run lint        # eslint
npm publish         # prepublishOnly runs build first
```

Build output:
- `dist/index.js` — ES module
- `dist/index.cjs` — CommonJS
- `dist/index.d.ts` — TypeScript declarations
- `dist/style.css` — all component and token CSS bundled

---

## Common mistakes to avoid

- Do not import a token CSS file that does not exist — this breaks the build silently.
- Do not add a component without exporting it from `src/index.ts`.
- Do not use `interface Props` — always use a named, exported interface.
- Do not hardcode colours or spacing — always use a `--token` variable.
- Do not add runtime dependencies (no `clsx`, no utility libraries) without explicit approval.
